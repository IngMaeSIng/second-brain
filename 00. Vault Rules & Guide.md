> How this vault works — for you and your AI agent. Read it once, then come back by section.

## 1. Purpose

This vault is a structured second brain for a single project or product. It holds the durable knowledge — the things you'll come back to — in plain text files that both you and your agent can read and write.

**Lives here:** strategy and goals · pricing, finance, legal · brand (voice, identity, logo, type) · product positioning and specs · design system and UX · engineering architecture and decisions · marketing channels and copy · research · decisions with rationale · meeting notes · people · companies · customer-ops procedures.

**Does NOT live here:** tasks and deadlines (a task tracker) · repo code and API docs (the product's GitHub repo) · financial documents like bank statements (an accounting service) · passwords and secrets (a password manager).

Rule of thumb: if it's worth coming back to later → vault. If it lives one day → chat. If it has a deadline → task tracker.

## 2. Quick start

**Install:** [Obsidian](https://obsidian.md/download) (free). For sync, optionally [GitHub Desktop](https://desktop.github.com/).

**Open:** Obsidian → *Open folder as vault* → select this folder.

**Enable community plugins** (Settings → Community plugins): **Dataview** (required — the indexes use it), **Templater** (for note templates), **Obsidian Git** (if you sync via GitHub).

**Settings worth setting:**
- Files & Links → Default location for new attachments → `16. Meta/Attachments`
- Files & Links → New link format → *Shortest path when possible*
- Editor → Default editing mode → *Live Preview*
- Templater → Template folder location → `16. Meta/Templates`

**Pin** `00. Dashboard` and this file so they're always one click away.

### Routing rules

The tricky calls — settle them once so notes land in the same place every time:

- **Inbox first.** Everything new lands in Inbox. Don't write straight into a domain unless you're sure where it goes.
- **Brand vs Design.** Brand = strategic identity, long-lived, applies everywhere. Design = product-specific UI/UX. Logo, palette, type → Brand. Components, screen patterns, UX flows → Design.
- **Brand vs Marketing.** Brand = identity and voice principles (the style guide). Marketing = execution and campaigns (short lifecycle). Voice doc → Brand. A campaign's copy → Marketing.
- **Product vs Engineering.** Product = WHAT and WHY (positioning, mechanics from the user's side). Engineering = HOW (architecture, integrations). Test: for the user, or for the developer?
- **Marketing research** → Research, tagged `domain/marketing`. Not Marketing.
- **Customer Ops vs Marketing.** Customer Ops = inbound procedures (how you answer, FAQ, refunds). Marketing = outbound (what you say publicly).
- **People vs Companies.** A person → Team. An organization → Companies.
- **Cross-cutting notes** (Decisions, Releases, Meetings) → tag with the `domain/X` they touch so that domain's index pulls them in.
- **One concept, one folder.** To reference from elsewhere, use `[[link]]` — never duplicate the file.

## 4. Naming conventions

**Folders:** `NN. Title Case` — e.g. `07. Product`. Sub-folder indexes get a decimal: `NN.MM Name Index.md`.

**Index files:** named with the folder's own number so they sort first inside the folder — `07. Product Index.md`. One index per domain.

**Notes:** `Title Case.md` — e.g. `Pricing experiment Q3.md`.

**Meetings & Decisions:** ISO date in front — `Weekly Sync 2026-06-17.md`, `2026-06-17 — pricing slug.md`.

**People:** `First Last.md`; on a clash, `First Last (Company).md`.

## 5. Tags

Five axes. All lowercase, singular, slash for hierarchy, hyphen for multi-word. In frontmatter without `#`, in body with `#`.

- **`domain/X`** — functional area (closed list): `team · business · brand · product · design · engineering · marketing · research · customer-ops`. Strategy, Releases, Decisions, Companies and Meetings are cross-cutting — tag them with the domain they touch.
- **`theme/X`** — topic (open vocabulary): e.g. `theme/pricing`, `theme/onboarding`, `theme/retention`.
- **`stage/X`** — lifecycle: `discovery · definition · design · build · launch · post-launch`.
- **`audience/X`** — your ICP segments.
- **`channel/X`** — comms channel: `telegram · linkedin · email · …`.

Hygiene: before a new tag, check the Tag pane. Once a month, kill singletons (tags used once) — rename into an existing one or remove.

## 6. Templates

Six templates in `16. Meta/Templates/`: **Note** (base), **Decision** (ADR), **Meeting**, **Person**, **Release**, **Post**.

Frontmatter properties:

| Property | Required | Used in |
|---|---|---|
| `type` | yes | all |
| `status` | yes | all except Meeting / Person |
| `tags` | recommended | all |
| `created` | auto | all |
| `updated` | manual | all |
| `root` | optional | all (link up to the domain index) |
| `related` | optional | all |
| `role` | yes | Person |
| `attendees` | yes | Meeting |
| `launch_date` | yes | Release, Post |
| `channel` | yes | Post |
| `revisit-by` | optional | Decision |
| `superseded-by` | conditional | Decision (when `status: superseded`) |

Status values: most notes `draft | active | archived` · Decision `proposed | accepted | rejected | superseded | on hold` · Release `draft | active | shipped | archived`.

**Supersession:** when a new decision replaces an old one — create the new one, set the old to `status: superseded`, add `superseded-by: "[[New Decision]]"` to the old.

## 7. Indexes

Every domain folder has one index (Inbox, Meta, Archive don't). An index has: a one-line description, **What lives here**, a manual **Pinned** section, a Dataview **Active** list (status draft/active), a Dataview **All** list, and **Materials**.

The Dataview sections update themselves. Maintain the Pinned section by hand on your weekly review.

## 8. Inbox processing

On your weekly review, clear Inbox. For each note: relevant? (no → delete) · assign a `type` · move to the right folder (see routing) · tag at least `domain/X` · rename in Title Case. After the pass, Inbox is empty.

## 9. Sync

**GitHub (recommended)** — version history and reliability. Create a private repo, clone it with GitHub Desktop, and let the Obsidian Git plugin auto-commit-and-sync every ~10 minutes (auto-pull on startup, push on backup). No terminal required.

**iCloud / any cloud drive (simplest)** — just keep the folder in a synced location. You lose version history but it works.

**Don't commit** workspace state and junk — see `.gitignore`. Do commit shared plugin settings so the vault behaves the same on every machine.

## 10. Collaboration (if you're not solo)

- Pull before a long edit; let auto-sync push after.
- Don't edit the same file at once.
- For a merge conflict: open the conflicted file, find the `<<<<<<<` / `=======` / `>>>>>>>` markers, resolve by hand, remove the markers, sync again.
- For breaking changes (renaming a folder or note), do it in one commit with a clear message, fix backlinks, and tell the others to pull before they start.

## 11. Working with your agent

The agent reads `CLAUDE.md` at the vault root — it's the condensed, agent-facing version of these rules (routing table + conventions). Keep the two in sync: when you change a rule here, reflect it there.

## 12. Updating this document

When you change a rule, update this file in the same commit as the structural change it describes.
